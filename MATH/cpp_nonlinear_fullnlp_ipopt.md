# `cpp_nonlinear_fullnlp_ipopt`

See [README.md](./README.md) in this folder for shared notation and the common path-following problem.

## 1. Controller Identity

- Profile ID: `cpp_nonlinear_fullnlp_ipopt`
- Solver backend: CasADi Opti with IPOPT
- Controller family: direct nonlinear model predictive control
- Intended comparison role: highest-fidelity direct nonlinear benchmark in the repository

## 2. Problem This Controller Solves

This controller solves a direct nonlinear optimal control problem at each control step. Unlike the RTI-SQP/OSQP profiles, it does not replace the dynamics with an affine stage model before solving.

Its implemented problem is

```text
min_{X,U} sum_{k=0}^{N-1} ell_k(X_k, U_k; ref_k) + ell_N(X_N; ref_N)
```

subject to

```text
X_{k+1} = f_d(X_k, U_k)
X_0 = x_meas
U_k in U
```

where `f_d` is the exact RK4-discretized nonlinear satellite model from the shared CasADi dynamics builder.

Physically, this controller is trying to find the best nonlinear trajectory and control sequence over the horizon that:

- pushes the satellite along the path through `v_s`
- minimizes cross-track and along-track mismatch
- keeps the body aligned with the path-based reference attitude
- damps translational and rotational motion near the endpoint
- limits wheel/jet effort and control chattering

## 3. Mathematical Transcription Used Here

### Decision Variables

The IPOPT controller uses full trajectory decision variables:

```text
X in R^{nx x (N+1)},   U in R^{nu x N}
```

with

```text
nx = 17,  nu = n_rw + n_thr + 1
```

and the same state/control interpretation as in the rest of the stack.

### Dynamics

The discrete dynamics constraints are imposed exactly through the shared RK4 discretization:

```text
X_{k+1} = f_d(X_k, U_k)
```

This is the main mathematical difference from the OSQP RTI profiles. There is no explicit affine approximation step and no QP subproblem at the user level.

### Cost Structure Actually Implemented

The implemented stage cost includes:

- contouring cost
- lag cost
- progress tracking cost
- progress reward term
- velocity-alignment cost
- attitude cost
- quaternion normalization penalty
- angular-velocity cost
- wheel/thruster control effort
- opposing-thruster penalty
- smoothness penalty `||u_k - u_{k-1}||^2` for `k > 0`

The terminal cost currently includes:

- terminal position pull
- terminal attitude cost

Important implementation caveat:

The current IPOPT transcription does **not** include all symbolic/shared terms that exist elsewhere in the stack. In particular, the current implementation omits:

- explicit terminal `s` cost using `Q_terminal_s`
- explicit terminal velocity cost using `Q_terminal_vel`
- online DARE terminal shaping
- linear thruster `L1` fuel-bias term
- explicit stage `s`-anchor term in the solve loop

So this controller is the most nonlinear in its dynamics transcription, but not the most feature-complete in its objective relative to the rest of the repository.

### Constraints

The implemented constraints are:

- exact nonlinear dynamics equality constraints
- initial-state equality
- wheel torque bounds in normalized coordinates
- thruster bounds `0 <= u_thr <= 1`
- virtual speed bounds `path_speed_min <= v_s <= path_speed_max`

The current IPOPT transcription does not add the acados-style hard delta-`u` constraints or terminal angular-rate bound constraints.

### Reference Construction

The controller builds a full horizon reference array from the current path estimate. The path reference at stage `k` is generated by advancing with the nominal path speed:

```text
s_k^ref approx s_now + k dt path_speed
```

and then computing:

- `p_ref,k`
- `t_ref,k`
- `q_ref,k`

with quaternion continuity logic that aligns body `+X` to the tangent.

## 4. How The Solve Proceeds Each Control Step

1. Form the augmented measured state including the current path coordinate `s`.
2. Build the full horizon reference trajectory from the stored path.
3. Set the initial-state, satellite-parameter, and reference parameters in the Opti problem.
4. Warm-start the NLP by shifting the previous state and control trajectories.
5. Call IPOPT on the full nonlinear program.
6. Extract the first control move from the solved control trajectory.
7. Advance `s` using the solved `v_s`, subject to the runtime mode logic.

This is a classic receding-horizon nonlinear MPC loop with shifted warm starts.

## 5. Why This Formulation Exists

This formulation exists because it provides the cleanest direct answer to the question:

What happens if the inspection-satellite controller solves the nonlinear problem itself instead of a sequence of linearized QP approximations?

That makes it the natural high-fidelity benchmark for the paper. It is useful when:

- solution quality matters more than raw solve speed
- the path geometry is strongly nonlinear
- the study wants a reference point for what the linearized controllers are approximating

The main engineering downside is runtime. Direct NLP solves are typically much more expensive and less predictable than one-shot sparse QPs.

## 6. Scientific Comparison Notes

### What makes it comparable

- Same physical state, control, and path-reference meaning
- Same shared nonlinear satellite model source
- Same broad path-following objective family
- Same fairness hashing mechanism for the shared baseline parameters

### What makes it not perfectly identical

- It solves the exact nonlinear dynamics, unlike the OSQP RTI family
- Its currently implemented objective omits some terms present in other controllers
- It uses IPOPT's nonlinear-program machinery rather than the structured Riccati/QP pipelines used by the other profiles

### Main paper caveat

This controller is the best direct nonlinear benchmark for dynamics fidelity, but it is not the canonical "same cost, different solver" benchmark because its implemented objective is not fully identical to the other stacks.

## 7. When This Controller Is Likely Best

Likely strongest when:

- maximum local model fidelity matters
- the compute budget is generous
- the study wants a benchmark close to full nonlinear path-following NMPC

Likely weakest when:

- the onboard processor must meet hard real-time deadlines
- solve-time predictability is important
- the mission needs the exact same terminal/control regularization features as the RTI stack
