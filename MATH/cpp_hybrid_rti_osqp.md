# `cpp_hybrid_rti_osqp`

See [README.md](./README.md) in this folder for shared notation and the common path-following problem.

## 1. Controller Identity

- Profile ID: `cpp_hybrid_rti_osqp`
- Solver backend: CasADi linearization in Python, sparse QP solve in C++ with OSQP
- Controller family: real-time iteration SQP with affine QP subproblem
- Intended comparison role: pragmatic real-time baseline that prioritizes robustness and continuity of control execution

## 2. Problem This Controller Solves

This controller tries to compute a physically feasible control sequence that moves the satellite forward along the inspection path while keeping translational and rotational errors small.

Conceptually, it solves

```text
min sum ell_k(x_k, u_k; p_ref,k, t_ref,k, q_ref,k) + ell_N(x_N)
```

subject to

```text
x_{k+1} = A_k x_k + B_k u_k + d_k
x_0 = x_meas
u_k in U,  x_k in X
```

where the affine stage model is built from nonlinear CasADi evaluation around the current trajectory iterate.

Physically, the optimizer is trying to:

- push the vehicle along the path through `v_s`
- reduce cross-track position error
- keep the body aligned with the path-based reference attitude
- keep angular velocity and control effort bounded
- avoid wasteful thruster usage and jittery control sequences

## 3. Mathematical Transcription Used Here

### Decision Variables

The QP decision vector stacks all horizon states and controls:

```text
z = [x_0, ..., x_N, u_0, ..., u_{N-1}]
```

with

```text
x_k = [p_k, q_k, v_k, omega_k, omega_rw,k, s_k]
u_k = [tau_rw,k, u_thr,k, v_s,k]
```

### Dynamics

The plant model used to generate the affine dynamics comes from the shared symbolic dynamics in [`../controller/shared/python/control_common/codegen/satellite_dynamics.py`](../controller/shared/python/control_common/codegen/satellite_dynamics.py). At runtime, the controller evaluates stage-wise discrete Jacobians and affine terms:

```text
x_{k+1} = A_k x_k + B_k u_k + d_k
```

where `A_k`, `B_k`, and `d_k` are generated by CasADi around the current trajectory iterate.

The state contains the virtual path coordinate `s`, so the optimizer is simultaneously choosing the physical actuation and how fast to move along the path.

### Cost Structure

The intended cost family is MPCC-style, but the actual QP cost assembled in the C++ runtime is the implementation that matters for this controller.

The major ingredients are:

- position/path term built from the active contouring and lag weights
- attitude tracking term about the path-derived reference quaternion
- translational velocity damping
- angular-rate damping
- `s` anchor term and optional terminal `s` pull
- wheel-torque and thruster effort
- smoothness penalty coupling `u_k` and `u_{k-1}`
- opposing-thruster pair penalty
- optional linear thruster `L1` penalty
- optional DARE-style terminal shaping

The runtime also supports mode-dependent cost scaling for `TRACK`, `RECOVER`, `SETTLE`, `HOLD`, and `COMPLETE`.

### Constraints

The main constraints are:

- affine dynamics equality constraints
- initial-state equality
- state bounds
- control bounds
- control-horizon tying constraints `u_k = u_{M-1}` for `k >= M`
- virtual path-speed bounds `path_speed_min <= v_s <= path_speed_max`

### Reference Construction

The path reference is built from the stored path using the current `s` estimate. The controller computes:

- `p_ref(s)` from the path interpolation
- a smoothed tangent `t_ref(s)` using lookahead/lookback logic
- a quaternion reference `q_ref(s)` that aligns body `+X` with the tangent and preserves frame continuity

This reference construction matters because the QP cost is always local to the currently generated path/attitude reference.

## 4. How The Solve Proceeds Each Control Step

1. Read the measured state and augment it with the current path coordinate `s`.
2. Warm-start the control sequence from the previously applied command if available.
3. Evaluate CasADi linearization stage by stage along the current trajectory.
4. If a stage linearization fails or returns non-finite values, keep going by reusing stale stage data instead of aborting the step.
5. Assemble the sparse QP in the C++ runtime.
6. Solve one OSQP QP.
7. Extract the first control move, update path progress bookkeeping, and store solve metadata.

The defining feature is step 4: the controller prefers continuity of operation over strict linearization integrity.

## 5. Why This Formulation Exists

This formulation exists because an inspection satellite needs a controller that is not only accurate, but dependable under real-time constraints.

The hybrid profile keeps the RTI-SQP structure, which is computationally cheap, while still using stage-wise nonlinear CasADi information when available. The stale-stage fallback is a practical engineering choice:

- it reduces hard controller dropouts
- it avoids sudden loss of actuation due to one bad stage evaluation
- it preserves real-time behavior when the model evaluation is imperfect

For a flight-feasibility discussion, this profile is attractive because it is the closest to a conservative real-time implementation choice.

## 6. Scientific Comparison Notes

### What makes it comparable

- Same state and control definitions as the rest of the stack
- Same broad path-following objective family
- Same shared baseline parameter block in fairness mode
- Same mission/runtime path-reference machinery as the other controller families

### What makes it not perfectly identical

- It solves a QP built from affine dynamics, not the full nonlinear program
- It uses the implementation-specific C++ QP cost, not the full symbolic cost in its most general form
- It tolerates stale Jacobian reuse, which changes the effective optimization problem when linearization fails

### Main paper caveat

This profile is best interpreted as the robust real-time RTI-SQP implementation, not as the purest nonlinear benchmark.

## 7. When This Controller Is Likely Best

Likely strongest when:

- onboard compute is limited
- strict real-time deadlines matter more than local optimality
- occasional model-evaluation failures must not interrupt control
- the mission values continuous safe operation over aggressive optimal refinement

Likely weakest when:

- the path geometry is highly nonlinear over the horizon
- the local affine approximation changes rapidly from stage to stage
- the experiment is trying to isolate solver accuracy rather than practical robustness
